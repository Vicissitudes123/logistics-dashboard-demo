# 物流部看板项目要求清单

> 本文件是项目后续设计、开发、验收和修改的统一依据。  
> 最后更新：2026-07-17（最终加固版）
> 当前状态：持续迭代

## 1. 项目定位

- 项目名称统一为“物流部看板”。
- 用于展示物流业务的管理指标、趋势、结构、异常和明细。
- 当前网页主要作为新版看板样板，供后续开发与验收参考。
- 页面应兼顾管理层快速浏览与业务人员进一步筛选分析。
- 不在看板正文中展示制作过程、数据质量整改过程或内部沟通来源。

## 2. 固定发布地址

- 正式演示地址固定为：https://vicissitudes123.github.io/logistics-dashboard-demo/
- 后续版本继续更新同一个 GitHub Pages 地址。
- 不使用需要登录 ChatGPT 的临时站点作为正式交付地址。
- 电脑和手机均应能通过普通浏览器直接访问。
- 发布后需要检查桌面端、手机端以及浏览器缓存下的显示情况。

## 3. 信息架构

管理总览之外，必须完整保留以下六个一级业务板块，名称不得随意修改：

1. 进口看板
2. 出口看板
3. 国内运输
4. 成本分析
5. 法检资料
6. 运费比价

如单个一级板块内容较多，可拆分为多个二级页面或页内子模块，避免所有内容堆叠在同一屏。

## 4. 视觉与布局

- 整体采用深色侧边栏、浅色内容区的深浅搭配。
- 风格应专业、简洁、稳重，有适度科技感，但避免过度炫光和明显的模板化、AI化效果。
- 科技感主要体现在数据呈现、交互反馈、动态过渡和信息层级，不只是深色背景或发光装饰。
- 卡片、图表和筛选区之间应保留足够留白，不能全部拥挤在一起。
- 颜色需要有稳定含义，同一指标在不同页面保持一致。
- 关键异常、下降、超期、未达标应有明确但不过度刺眼的提醒色。
- 侧边栏必须支持收起和展开。
- 侧边栏控制按钮应位于页面左侧边缘的垂直居中位置，不能占用管理总览标题区域。
- 页面应适配常见公司电脑分辨率，并兼顾手机浏览。

## 5. 时间筛选与对比

- 支持 2024、2025、2026 年数据。
- 日期展示统一为清晰、一致的格式；年月建议统一使用“YYYY年MM月”，完整日期建议统一使用“YYYY-MM-DD”。
- “本月”应基于当前系统日期；如果当前月份无数据，应明确提示或自动定位到最近有数据月份，不能静默产生误解。
- YTD 表示“本年截至当前月份的累计数据”，界面中应提供中文解释。
- 年、月、时间范围选择必须真实联动本页全部适用的指标、图表和明细。
- 图表横轴必须明确显示年份和月份，不能只写“7月”而无法判断年份。
- 支持选择多个时间段、多个年份或多个月份进行比较。
- 对比指标选择后，应能明显改变图表或汇总结果。
- 时间和指标选择区都应提供清晰的“全选”和“全不选”按钮。

## 6. 图表与数据呈现

- 优先呈现管理者真正关心的结论：现状、变化、原因、异常和需要关注的对象。
- 折线图、柱状图、饼图、环形图等应有明确标题、单位、图例和时间范围。
- 数据标签在不重叠、不拥挤时应直接显示。
- 如果直接显示会重叠，则改为悬停显示，不能为了展示全部数字而降低可读性。
- 直接显示数据标签后，悬停提示中不重复堆叠同一数字。
- 折线、数字、图例和坐标轴之间不能互相遮挡。
- 饼图或环形图必须清楚说明各颜色代表的类别，并显示合理的数值或占比。
- 类别较多时优先使用排序条形图、Top N 加“其他”等方式，避免出现无法辨认的多色饼图。
- 图表切换、筛选和页面进入动画应流畅自然，避免突然跳变。
- 动画以辅助理解为目的，不能拖慢页面或影响操作。
- 不应在管理看板中展示仅供开发人员查看的“数据质量提示”；相关问题应在项目问题清单或整改建议中单独记录。

## 7. 筛选与交互

- 所有“全选”和“全不选”必须呈现为明显可点击的按钮，不能只是普通文字。
- 明细表每个可比较字段应提供类似 Excel 的筛选功能。
- 字段筛选至少支持：
  - 直接勾选一个或多个值；
  - 搜索或手动输入关键字；
  - 全选；
  - 全不选；
  - 清除当前字段筛选；
  - 应用筛选。
- 筛选结果应同步影响汇总指标、图表和明细表。
- 用户应能自行选择需要放在一起比较的字段和指标。
- 明细区应提供指标/字段选择面板，并包含全选和全不选。
- 当前选中的年份、月份、指标和筛选条件应有明显状态反馈。
- 页面刷新后可尽量保留合理的默认状态，不能出现空白但无说明的页面。

## 8. 明细表要求

- 每个一级业务板块底部应保留可追溯的明细数据。
- 默认只展示最有用的核心字段，其他字段由用户自行选择。
- 支持横向滚动，但关键字段应尽量固定在左侧。
- 支持字段筛选、排序、搜索和组合比较。
- 表头、单位和日期格式必须统一。
- 表格不能一开始就把所有字段和所有数据全部堆在页面中。
- 可根据数据量提供分页、每页条数或按需加载。
- 明细数据与上方汇总口径应一致。

## 9. 数据范围与真实性标识

- 优先使用现有提供的数据文件和看板资料。
- 当前样板纳入 2024、2025、2026 年数据。
- 数据缺失时允许为展示交互而补充模拟数据。
- 所有模拟、补齐或推演数据必须明确标注“演示数据”或“非真实数据”。
- 不得让使用者把模拟数据误认为正式经营数据。
- 样本量较少不影响样板制作，但正式开发时需要重新校验指标口径和数据完整性。
- 成本分析目前以已获得的数据为基础制作，后续有新增历史数据时继续补入。

## 10. 六大板块的核心关注方向

### 10.1 进口看板

- 票数、净重、费用、单位成本、查验、整改、时效和供应商/国家分布。
- 支持海运、空运、DG/NDG、贸易条款、供应商等维度分析。
- 关注查验达标率、整改情况、异常月份和相关明细。

### 10.2 出口看板

- 票数、净重、费用、单位成本、国家分布、贸易条款和 FCL/LCL。
- 展示月度趋势、同比/环比、船期晚开原因及重点国家或路线。
- 关注成本节约、费用变化和出口业务明细。

### 10.3 国内运输

- 订单数、运单数、重量、费用、单位成本、快递与进仓业务。
- 展示运输状态、准时性、异常、客户/目的地分布和业务流向。
- 管理监控、业务情况分析和明细应形成清晰的层级。

### 10.4 成本分析

- 覆盖进口、出口、国内运输的成本趋势和结构。
- 支持成本、票数、净重、成本/千克、成本/票等指标切换。
- 支持同比、环比、DG/NDG、贸易条款、FCL/LCL、路线等维度。
- 成本变化应尽量显示主要原因，而不只是罗列数字。

### 10.5 法检资料

- 支持单据、产品、发货方、IATA 代码等资料查询。
- 筛选项、表头和中英文命名应统一。
- 资料型页面以快速查询、准确定位和清晰明细为主，不必强行加入无意义图表。

### 10.6 运费比价

- 支持不同承运商、运输方式、路线和时间的报价比较。
- 应清楚展示价格差异、重量区间、附加费、适用条件和推荐结果。
- 保留历史报价及报价申请查询能力。
- 比价结果需要说明比较口径，避免不同条件的价格被直接错误比较。

## 11. 管理总览

- 管理总览是入口，不计入六大业务板块数量。
- 应突出少量核心指标、关键趋势、主要异常和需要关注的事项。
- 管理总览中的时间筛选必须联动下方图表。
- 指标说明应通俗，避免只出现未解释的英文缩写。
- 不展示开发人员内部使用的数据质量检查提示。
- 总览应能进入六大业务板块继续查看原因和明细。

## 12. 性能与兼容性

- 页面切换和筛选反馈应顺畅。
- 避免一次渲染过多图表、标签和明细行。
- 主要功能不依赖 ChatGPT 登录。
- GitHub Pages 版本应为可独立运行的静态网页。
- 修改完成后至少检查：
  - 六大板块能否正常切换；
  - 年月筛选是否联动；
  - 全选/全不选是否有效；
  - 图表是否重叠；
  - 明细筛选是否有效；
  - 侧边栏能否收起/展开；
  - 手机和电脑是否可访问；
  - 控制台是否存在明显错误。

## 13. 后续更新规则

- 新增要求时，优先更新本文件，再修改网页。
- 修改要求时，在原条目上更新，避免保留互相冲突的旧要求。
- 完成某项功能后，可在对应条目或项目进度文件中标注完成状态。
- 如果新要求与现有要求冲突，应先确认优先级。
- 正式发布前，根据本清单逐项验收。

## 14. 第六版补充要求（2026-07-17）

### 14.1 大屏与视觉重点

- 页面主要用于公司大屏展示，关键标题、坐标轴、图例、数据标签和流程文字必须保持远距离可读。
- 版面不能出现“大卡片、小图表”或上下留白过多的情况；趋势图应充分利用卡片宽度和高度。
- 预计到常州、ATA/ETA + 8WD、超期、临期、异常、待办、国内运输及时率与异常率、供应商履约和报价推荐属于重点管理信息，应形成明显的视觉层级。
- 红色只表示已超期或严重异常，橙色表示临期或待处理，蓝绿色表示正常或完成；不能让所有卡片使用相同视觉权重。
- 管理总览中的重点事项应支持直接进入对应业务子页面。

### 14.2 图表自动标签与历史数据扩展

- 折线图应根据图表宽度、系列数量、数据点数量、文字实际宽度和标签碰撞情况自动决定标签策略。
- 标签空间充足时持续显示；可能重叠时隐藏普通标签，并在鼠标移动到月份或数据点时显示该时点的完整数值。
- 自动判断不能针对当前演示数据穷举，应能适配后续补入的 2016 年至今历史数据。
- 多年份对比时横轴或悬停信息必须明确年份和月份，颜色不足时不得简单循环造成辨认困难。
- 纵轴精度应根据数据跨度自动选择，不能出现不同刻度被四舍五入为同一数字。
- 图表卡片应按信息密度使用整行或合理分栏，避免关键趋势缩在页面角落。

### 14.3 地图

- 使用真实可缩放、可拖拽地图；气泡点击后应可靠聚焦并打开指标信息。
- 首次加载、筛选、切换子页面、侧边栏收起展开和全屏切换后都应重新计算地图尺寸。
- 地图应按当前可见气泡自动适配范围，恢复全局也应回到全部可见点范围，不能裁掉目的地。
- 标题、缩放按钮、弹窗和全屏按钮不能互相遮挡；底图加载失败时应提供重试或备用底图。

### 14.4 订单检索和流程

- 单票时间线支持输入完整或部分订单号，并提供自定义可搜索下拉框。
- 输入框已有内容时，点击右侧箭头仍应显示当前范围内的全部订单；输入文字后可过滤选项。
- 不同订单必须绑定不同节点、日期、状态和预测结果，不能只改变订单号而复用同一时间线。
- 甘特图在常用桌面和大屏宽度下完整横向显示，不依赖底部滚动条；阶段、等待时间和超期区段必须清楚。

### 14.5 明细表

- 保留全局搜索，同时每个字段提供独立的 Excel 式筛选。
- 分类字段支持搜索、多选、全选、全不选、清除和应用；文本字段支持包含、等于、为空；数值和日期字段支持比较与区间。
- 不同字段筛选按“并且”组合，同一字段多值按“或者”组合；当前条件应以可移除标签显示。
- 恢复自定义字段功能：显示/隐藏、全选、全不选、恢复默认，并支持调整字段顺序和固定关键列。
- 筛选、排序、字段选择和全局搜索应适用于六大板块的明细，并尽量同步影响上方汇总与图表。
- 数据量扩大后应提供分页或按需渲染，不能因录入多年数据而一次渲染全部记录。

### 14.6 时间、运输方式与导出

- 需要同时支持全部、仅海运、仅空运，并使适用的 KPI、图表、地图和明细同步变化。
- MTD 仅放在适合观察当月进度的位置，YTD用于年度累计；二者均需中文解释并按可用数据日期计算。
- PDF 保留一个统一入口，可选择六大板块中的任意组合，支持全选、全不选和调整顺序，并生成可保存的 PDF 文件。
- PDF 中的图表、地图、表格和重点状态应保持可读，不能因分页被随意截断。

### 14.7 数据与性能

- 修复重复订单、混合日期格式、演示汇总与明细口径不一致等问题。
- 当前仍可使用匿名演示数据，但数据结构不得依赖固定年份、固定行数或特定示例值。
- 表格搜索和筛选不应无必要地销毁并重建地图；多年数据场景下仍需保持交互流畅。
- 外部地图、Logo和PDF组件加载失败时需要明确降级提示，不能只留下空白区域。

## 15. 最终加固版补充要求（2026-07-17）

- 完整保留第六版已经实现的逐字段筛选、自定义字段、订单检索、流程、甘特图、图表自动标签、海空运切换和PDF组合选择，不得因兼容性修复而回退。
- 地图必须识别“程序已加载但底图瓦片为空”的情况，按超时或连续错误自动切换多个底图来源；全部网络底图失败时，使用同站点的真实国家边界数据兜底，不能继续显示空白背景。
- 地图、PDF组件和官方Logo应尽量使用仓库内公开静态资源，减少企业网络对外部CDN的拦截风险。
- 订单下拉列表必须支持鼠标滚轮、触控板、触摸拖动、可见滚动条和键盘上下选择；根据视口剩余空间自动向上或向下展开。
- 明细搜索、逐字段筛选、排序、分页和字段显示调整时应保留当前地图实例，不能无必要地销毁并重新加载底图；上方汇总仍需同步刷新。
- PDF板块选择器应明确区分“当前子页面”和“板块全部子页面”，并在导出后恢复用户原来的页面与子页面状态。
- MTD、YTD和最新数据月份提示必须根据系统日期、数据截止日期和所选年份动态计算，不得硬编码当前月份。
- 可选年份应由数据范围生成，不逐项写死年份数组，为后续扩展至2016年至今保留统一入口。
